01 📡 多源信息采集

内置三个权威行业 RSS 源（36氪、雷锋网、iThome）作为基础设施层，同时支持手动添加任意 RSS Feed。APScheduler 定时调度，默认每 6 小时自动轮询所有活跃源。

核心逻辑：feedparser 逐源解析 → 提取标题/正文/发布时间 → URL 级去重（`WHERE source_id = :sid AND url = :url`）→ 写入 `raw_articles` 表（is_processed=false）→ 更新源的最后抓取时间。采集和报告生成是独立的两个步骤，互不阻塞。

02 🔍 主题级关键词匹配与多源关联

这是智闻区别于普通 RSS 阅读器的核心设计。不同于所有文章扔进一个池子，智闻采用「主题 → 关键词 → 源」的三层映射：每个主题可配置专属关键词和排除词，并关联一个或多个新闻源。

匹配逻辑：遍历主题的所有关联源 → 对每篇新文章做 `title + content + summary` 全文模糊匹配 → 命中任一关键词即标记 `is_processed=true` → 写入 `topic_sources` 关联表。排除词优先级高于关键词，命中排除词的文章直接跳过。

重要设计决策：即使 DeepSeek API 调用失败，也要将 `is_processed` 置为 true，防止失败文章被无限重试。匹配锚点用 `published_at`（而非 `fetched_at`），保证周期性报告能覆盖到正确的时间范围。

关键词过滤排除词三层映射多源关联published_at锚点防重复处理

03 📅 日历周期 + 向上兼容调度 + 重叠守卫

这是智闻报告系统的核心创新点。报告的"时间窗口"不使用简单的「过去N天」，而是使用自然日历周期：

日报：今日 00:00 UTC → 当前（范围：1天）
周报：本周一 00:00 UTC → 当前（范围：累计本周）
月报：本月1日 00:00 UTC → 当前（范围：累计本月）
季报：本季首月1日 00:00 UTC → 当前
年报：本年1月1日 00:00 UTC → 当前

之所以这样设计，是因为「过去7天」在周三生成的周报覆盖的是上周三到这周三的碎片区间，而CEO需要的是「本周总结」这个语义正确的范围。日历周期确保每周一、每月1号、每季首日都能产出语义正确、范围清晰的报告。

向上兼容调度：配置为「周报」的主题，自动也会按时生成月报、季报、年报。层级关系：`daily → weekly → monthly → quarterly → yearly`。配置日报的主题会生成所有5种周期，配置季报只生成季报+年报。

日历重叠守卫：每月1号恰好是周一时，日报、周报、月报三个任务同时触发。智闻在 `_run_report` 的首行检查：每月1号跳过日报+周报，只保留月报；周一（非1号）跳过日报只保留周报。优先级：monthly > weekly > daily。

日历周期非滑动窗口向上兼容调度重叠守卫语义正确5级周期

04 🤖 AI 深度分析引擎（SWOT + 风险 + 机会 + 重试）

这是智闻的大脑。匹配到的文章按以下流程处理：

  1. 文章快照封装：取匹配文章（上限50篇），构建 JSON 快照（id/title/summary/url/published_at），嵌入报告 content 字段。>30篇时只传标题+摘要以节省 token。
  2. AI Prompt 构建：基于报告类型动态调整分析深度。日报用「快速分析」（每项100-150字），周报用「深度综合分析」（每项200-300字），月/季/年报用「深度战略分析」。
  3. DeepSeek 调用：调用 `chat_with_deepseek()`，要求返回结构化 JSON（s/w/o/t + risks[] + opportunities[] + risk_level + key_trends + action_items）。
  4. Token 预算设计：这是开发过程中踩过的最大的坑。deepseek-v4-flash 是推理模型（Reasoning Model），每次回答前先做链式思考（`reasoning_content`），思考过程消耗大量 token。79-94篇文章的周报 prompt 下，3072 的 `max_tokens` 被思考过程全部吞掉，留给实际 JSON 输出的 token 为 0 → AI 返回空内容 → 3次重试全部失败。修复方案：所有 token 预算翻倍：日报8192、周报12288、月/季/年报16384。
  5. 三次重试 + 退避策略：首次调用用默认参数，后续重试用 httpx 直连 API，`temperature` 逐次+0.1，`max_tokens` 翻倍，system prompt 追加「请务必输出有实质内容」指令。
  6. 风险等级标准化：AI 有时输出「整体风险等级：高」，有时输出 `"high"`，有时输出「高」。后端 `_normalize_risk_level()` 统一清洗：去掉前缀、映射中英文、fallback 到「中」。
  7. Has_substance 守卫：如果所有 SWOT 字段 + 风险列表 + 机会列表均为空，将 status 设为 `draft`（不入库展示、不推送），而不是 `generated`。这是防止空报告推送的关键安全阀。

重试循环的 bug 修复：重试时如果 httpx 抛出异常，`response` 变量未被更新，后续循环一直使用第一次的错误值。修复：每个异常分支强制赋值 `response = f"（重试异常）{str(retry_err)}"`。

05 📊 报告存储 + 文章快照 + 回收站

报告不是简单的文本文件，而是结构化数据对象：

reports 表字段：
id | topic_id | report_type | title
summary | content | swot
risk_level | risk_items | opportunities
push_time | status | feishu_doc_token | feishu_msg_id
is_deleted | deleted_at | created_at | updated_at

content 文章快照：报告自包含源文章信息（标题、摘要、URL），即使源站删除也能查阅。快照结构：`{"articles": [...], "time_range": {"start": ..., "end": ..., "label": "第23周"}, "key_trends": ..., "action_items": [...]}`。

空周期回溯：如果当前周期（如本周）无匹配文章，自动回溯最多7天查找。找到的文章在报告标题加「含回顾」后缀，保证报告不出现空内容。

重要设计决策：refresh pipeline 绝对不删除旧报告！之前的实现有一个 `DELETE FROM reports WHERE topic_id IN (...)` 步骤，每次手动刷新都会清空所有历史报告。修复：只 INSERT 不 DELETE，前端显示最多8条足够了。

回收站安全设计：软删除（`is_deleted=1`）→ 回收站列表 → 可恢复（`is_deleted=0`）→ 或永久删除（`DELETE WHERE is_deleted=1`）。所有增删操作都二次确认弹窗。

文章快照JSON结构化空周期回溯不删除旧报告软删除回收站

06 📨 三条推送路径 + 安全推送到飞书

智闻的报告推送不是单一路径——这是开发过程中发现的一个重要架构教训。系统实际存在三条独立的推送代码路径，它们共享推送逻辑但不共享代码：

路径1（Auto-push）：`reports.py` 的 `_run_report_generation` 尾部——AI 报告生成成功后，自动检查 `topic_push_configs` 中是否有 Webhook URL 或飞书 OAuth 配置，有则推送。
路径2（手动推送）：`report_push.py` 的 `POST /api/v1/reports/{id}/push`——用户在报告页面手动点击「推送到飞书」时触发。
路径3（CEO顾问生成+推送）：`ceo_advisor.py` —— CEO顾问在对话中直接生成报告并推送。

关键 bug 修复：最初只在路径1添加了风险识别和机会发现模块，路径2和3没有。CEO 顾问推送的报告缺少「风险识别」和「机会发现」卡片模块→用户反馈后在三条路径全部同步添加。

推送卡片内容：飞书 interactive 消息卡片，包含：📋摘要 → 📊 SWOT四象限 → 🔍风险识别（🔴🟡🟢图标 + 最多3条）→ 🌟机会发现（最多3条）→ 📰源文章快照（前10篇带链接）。卡片头部颜色随风险等级变化（红/黄/绿）。

双通道容错：OAuth API 推送和 Webhook 推送并行，任一路径失败不影响另一路。OAuth 需要飞书自建应用的 `LARK_APP_ID` + `LARK_APP_SECRET`，Webhook 只需一个 URL。

07 🔧 精细化控制：三个独立开关 + 手动采集器

每个主题的自动化不是「开/关」二元控制，而是三个独立开关：

📥 数据采集：控制 APScheduler `collect_{topic_id}` 任务的启停。开 = 每6小时抓取 RSS。
📊 报告生成：控制 `report_{type}_{topic_id}` 定时任务的启停。开 = 按 push_cycle 生成报告。
📨 飞书推送：控制 `topic_push_configs.feishu_push_enabled`。开 = 报告生成后自动推送到飞书。

每个开关通过 `PUT /api/v1/automation/topics/{id}/config` 独立更新，APScheduler 实时生效无需重启服务。数据库持久化（`scheduled_jobs` 表），应用重启后自动恢复。

手动采集器：除了定时调度，还提供「一键刷新」入口（`POST /api/v1/reports/refresh`）。用户可选择指定主题手动触发：采集 RSS → 关键词匹配 → 生成 AI 报告。修改关键词后可立即重跑验证效果，无需等下一轮调度。

三独立开关APScheduler实时生效数据库持久化手动采集器一键刷新

Capabilities

核心优势

普通的信息聚合工具 ≠ 智能情报系统。智闻的每一项能力都围绕「决策效率」设计。

📚

多主题并行追踪

支持创建任意数量的行业主题，每个主题独立配置关键词、新闻源和报告周期，互不干扰。可同时追踪「机器人出海」「AI最新动态」「新能源汽车」等行业。

📅

5级日历周期

日报/周报/月报/季报/年报，使用自然日历范围，自动错开重叠周期。每月1号只生月报不生成日报和周报，避免重复。

🎯

AI SWOT 分析

DeepSeek 驱动的结构化分析引擎，输出优势/劣势/机会/威胁四象限，附带风险清单和商业机会清单，每项都引用源文章支撑。

🔔

主动推送到飞书

报告生成后自动推送到企业飞书群，CEO 无需登录系统即可在聊天中阅读。卡片包含摘要、风险等级标识和源文章链接。

⏰

精细化自动调度

每个主题三个独立开关：数据采集、报告生成、飞书推送。可独立启停，APScheduler 实时生效，无需重启服务。

🔒

数据安全可靠

单机 SQLite 部署，无需外部数据库。每日自动全量备份，保留30天历史。Git 实时推送代码到 GitHub，双重保障。

⚡

一键手动刷新

支持手动触发全量刷新：采集最新 RSS → 关键词匹配 → 生成 AI 报告。修改关键词后可立即重跑，无需等待定时调度。

🔄

空周期自动回溯

当前周期无数据时自动回溯7天查找文章，保证报告不出现空内容。回溯数据在报告中标注「含回顾」标记。

🔐

多组织隔离

基于 org_id 的数据隔离，每个注册用户自动分配独立组织。不同企业的主题、报告、推送配置完全隔离。
